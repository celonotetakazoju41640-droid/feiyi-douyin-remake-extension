# 飞蚁抖音复刻路线图

这份文档只说 3 件事：

1. 现在这版已经做到哪。
2. 下一阶段最该先做什么。
3. 哪些东西还不能假装成“已经完成”。

## 当前版本定位

当前仓库已经是一个可加载、可使用、可迭代的 Chrome 扩展原型，重点不是直接跑真实视频生成，而是把“对标账号蒸馏 -> 参考视频拆解 -> 复刻包整理 -> 批量任务输出”这条链路先打通。

当前最成熟的部分是：

- TikTok 公开主页样本蒸馏
- 对标账户模板沉淀
- 样本筛片和重蒸馏
- 复刻包 / Clipcat 指令 / 批量任务输出
- TikTok / 抖音双平台入口统一

当前还处于“半自动工作台”阶段，不是“全自动成片系统”。

## 下一阶段优先级

### P1：抖音主页自动蒸馏第二层

当前已完成到这里：

- 抖音站点权限
- 抖音公开主页链接识别
- 公开样本链接抽取
- 模板字段回填
- 复用现有筛片台入口
- 第二层增强已接入：
  - 优先解析页面脚本数据
  - 再合并 DOM 卡片结果兜底
  - 扫描时允许多轮重试，优先保留样本质量更高的一轮结果
  - 在部分真实主页上，已经能从可见卡片补到封面、点赞和更干净的标题
  - 已开始对前几条缺失样本补抓公开视频详情页，回填时长、评论和分享
  - 当前详情页补数已改成按缺失程度和样本价值优先补，并在空白页时自动重试一次
  - TikTok / 抖音样本里的收藏数现在都会统一保留到样本摘要、筛片卡片和候选导出里

下一步目标：

- 继续提高样本文案、封面、时长、互动数据的准确率。
- 从“第二层增强可用”继续提升到“样本质量足够稳定”。

下一步最稳妥拆法：

1. 继续深挖抖音页面 JSON / SSR 数据的字段覆盖率。
2. 进一步提升样本标题和封面命中率。
3. 尽量补齐时长、点赞、评论、分享等互动数据。
4. 再考虑更深的账号全页蒸馏。

完成标准：

- 样本标题不再主要依赖页面 SEO 锚文本兜底。
- 样本封面命中率明显提高。
- 至少能为大部分样本补齐更可靠的时长或互动数据。

### P2：本地服务对接收口

当前前端已经能把任务送到本地 `4328` 服务，但仓库里还没有把这部分完整纳入。

下一步建议：

- 单独补服务端目录或把服务端仓库关系写清楚。
- 明确当前服务端只负责接任务，还是已经能真正分发到上游视频接口。
- 给失败状态、重试状态、队列状态补更清楚的返回口径。

完成标准：

- 仓库里能清楚说明服务端在哪。
- 前端能看到服务状态，不只是“可用 / 不可用”。
- 批量任务发送后的反馈更明确。

### P3：真实任务运行前的安全层

这块现在故意没打通到底。

下一步如果要接真实视频生成，必须先补：

- 资源消耗规则
- 失败回退规则
- 任务状态记录
- 成本留痕
- 不暴露密钥的服务端入口

完成标准：

- 前端不直接调用真实密钥
- 每次任务都有状态
- 能区分“提交成功”“平台失败”“需要重试”

## 暂不建议现在就做的事

这些不是不能做，而是不该抢在前面：

- 直接接入真实视频大批量实跑
- 一上来做抖音深翻全站内容
- 顺手把整个工作台大重构
- 为了“统一”重写现有 TikTok 蒸馏链路
- 在前端保存任何 API Key

## 当前已知边界

### 能做

- TikTok / 抖音公开主页样本蒸馏
- 模板沉淀
- 复刻包生成
- 批量任务整理
- Clipcat 指令整理

### 不能算已完成

- 抖音高准确率主页蒸馏
- 真实视频生成闭环
- 真实成本 / 额度控制
- 账号登录态抓取
- 私有内容抓取

## 建议开发顺序

后续继续开发时，建议按这个顺序推进：

1. 抖音公开主页样本蒸馏
2. 抖音样本质量提升
3. 本地服务状态和结果反馈补全
4. 真实任务运行前的安全层
5. 再考虑真实视频生成接通

## 每次继续开发前要先核对

- `README.md` 里的能力描述是不是还和代码一致
- `manifest.json` 的权限是不是和平台能力一致
- `src/workspace.js` 的平台切换逻辑是不是还一致
- `tests/remake-core.test.mjs` 有没有同步补测试
- 当前提交的工作区是不是干净

## 当前仓库目标一句话

先把“能稳定复刻、能沉淀模板、能整理批量任务”做到扎实，再去接“真实自动生成”。
