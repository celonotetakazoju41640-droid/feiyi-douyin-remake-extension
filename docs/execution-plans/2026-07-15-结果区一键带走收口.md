# 2026-07-15 结果区一键带走收口

## 用户原始需求摘要

用户要的是一条龙，不是让用户自己判断先复制、再导出、再下载哪一个按钮。

## AI 对需求的理解

当前结果区虽然已经有：

- 复制结果包
- 故事版主按钮
- Markdown / JSON 下载
- 批量导出项目

但这些动作还是分散的。对于普通用户来说，仍然需要自己理解“下一步该点哪一个”，这不算真正的傻瓜式收口。

## 当前假设

- 结果区应当存在一个真正的主动作。
- 如果故事版还没完成，主动作应该先引导或直接触发故事版补齐。
- 如果结果已经齐全，主动作应该直接完成“复制结果包 + 导出文件 + 下载图片”。

## 计划修改范围

- `src/workspace.html`
- `src/workspace.js`
- `tests/extension-shell.test.mjs`
- `README.md`
- 本执行计划文档

## 明确不修改的范围

- 不改蒸馏算法
- 不改抖音 / TikTok 提示词语言逻辑
- 不改蒸馏管理页结构
- 不碰 `scripts/runtime/`
- 不碰 `src/workspace-flow-test.html`
- 不处理无关的 `src/workspace.css` 既有脏改

## 执行步骤

1. 先补测试，证明结果区存在真正的一键带走主按钮。
2. 修改结果区：
   - 故事版未完成时：显示“先生成故事版再带走”或类似主按钮
   - 结果齐全时：显示“一键带走全部结果”
3. 主按钮执行：
   - 复制结果包
   - 下载导出文件
   - 下载故事版图片
4. 同步 README。
5. 跑相关检查命令并完成 Git 闭环。

## 验收标准

- 结果区存在一键带走主按钮。
- 用户不需要自己判断复制、导出、下载的顺序。
- 故事版未完成时，主按钮会优先补齐故事版流程。
- `node --check src/workspace.js`
- `node --test tests/extension-shell.test.mjs`
- `git diff --check`
- `git diff --cached --check`

## 风险和需要注意的事项

- 主按钮不能和已有故事版主按钮、复制结果包按钮打架。
- 文案必须跟真实状态一致，避免出现“一键带走”但实际上结果还没齐的误导。

- 故事版未完成时，主按钮会先引导到“先生成故事版再带走”。
- 结果齐全时，主按钮会直接执行：
  - 复制结果包
  - 下载导出文件
  - 下载故事版图片
- README 已同步更新当前主流程说明。
