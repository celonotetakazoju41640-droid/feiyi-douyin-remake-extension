# 2026-07-15 一键带走自动串行收口

## 用户原始需求摘要

用户要的是一条龙，不是点击一次故事版、再点击一次导出。

## AI 对需求的理解

当前结果区已经有 `一键带走全部结果` 主按钮，但在故事版未完成时，这个按钮只会先去跑故事版，然后停住，用户还要再点一次。

这不符合真正的一次点击串到底。

## 当前假设

- `runStoryboardFlow()` 已经会自动轮询到终态或超时。
- 所以主按钮在故事版未完成时，可以先调用故事版流程，再在故事版已就绪时自动继续复制和导出。

## 计划修改范围

- `src/workspace.js`
- `tests/extension-shell.test.mjs`
- `README.md`
- 本执行计划文档

## 明确不修改的范围

- 不改蒸馏算法
- 不改抖音 / TikTok 提示词语言逻辑
- 不改蒸馏管理页结构
- 不碰 `scripts/runtime/`
- 不碰 `src/workspace-flow-test.html`
- 不处理无关的 `src/workspace.css` 既有脏改

## 执行步骤

1. 先补测试，证明一键带走会在故事版完成后自动继续整理结果。
2. 修改一键带走逻辑：
   - 故事版未完成时先跑故事版
   - 故事版就绪后自动继续复制和导出
3. 同步 README。
4. 跑相关检查命令并完成 Git 闭环。

## 验收标准

- 用户点击一次 `一键带走`，如果故事版能在当前轮询内完成，就会自动继续复制和导出。
- 只有故事版仍未完成时，流程才会停在故事版阶段。
- `node --check src/workspace.js`
- `node --test tests/extension-shell.test.mjs`
- `git diff --check`
- `git diff --cached --check`

## 风险和需要注意的事项

- 不能出现重复导出或重复提示。
- 文案必须说明是“故事版已就绪，继续自动整理结果”。

## 执行后的实际结果

- 一键带走在故事版未完成时，不再只停在故事版阶段。
- 如果故事版能在当前轮询里完成，当前会自动继续：
  - 复制结果包
  - 导出文件
  - 下载故事版图片
- 只有故事版仍未完成时，流程才会停在故事版阶段。
- README 已同步更新当前真实行为说明。
