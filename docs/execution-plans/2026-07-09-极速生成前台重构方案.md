# 飞蚁抖音复刻极速生成前台重构方案

## 这次为什么要重来

当前页面虽然已经做过几轮简化，但问题还很明显：

- 首页默认还是带着“模板 / 蒸馏 / 高级设置 / 结果细节”的后台思维。
- 普通用户一打开，仍然需要理解太多系统内部概念。
- 页面虽然变短了一些，但上半屏的主任务仍然不够聚焦。
- 产品结构还没有真正切成“普通用户前台”和“官方运营后台”两套。

这次不再把重点放在“继续润色当前页面”，而是直接把产品重新定义成两套入口。

## 用户原始需求摘要

用户希望：

- 不要继续靠想象改页面。
- 要参考成熟工具类产品的完整思维方式。
- 希望有从需求、设计、开发到验收的正规流程。
- 希望能按多 agent 明确分工，而不是所有事情混在一个思路里。

## 这次产品定义

### 1. C 端前台

目标只有一个：

- 用户上传素材，快速拿到可直接提交的视频提示词。

这个入口默认只暴露用户能立即理解的动作：

- 选择平台
- 上传产品图
- 可选上传模特图
- 可选贴一个对标链接
- 补一句风格要求
- 生成提示词

系统自动做这些隐藏动作：

- 选择默认模板
- 判断平台对应语言
- 尝试参考对标主页风格
- 自动提炼卖点草稿
- 失败时自动回退到默认模板

### 2. 官方运营后台

目标是沉淀和维护风格能力，不是给普通用户直接操作。

这个入口保留：

- 模板管理
- 主页蒸馏
- 样本筛片
- 重蒸馏
- 高级提示词控制
- 批量任务和本地服务

## 这次重构的核心原则

### 原则 1：普通用户只做主任务

普通用户首页不出现系统内部概念，不要求理解模板、DNA、样本筛片、蒸馏规则。

### 原则 2：复杂度后置

复杂能力不删除，但必须退到折叠区或单独后台入口。

### 原则 3：系统自己承担判断

凡是平台、模板、语言、默认风格、降级策略这些逻辑，优先由系统自动完成。

### 原则 4：首屏只放最少内容

用户打开页面后，首屏必须一眼看懂：

- 我要传什么
- 可以不可以直接生成
- 现在点哪里

## 多 agent 分工

### Agent 1：产品与信息架构

负责：

- 定义前台和后台的边界
- 删除普通用户不需要看到的信息
- 明确每个模块的职责

产出：

- 前台页面结构
- 后台折叠区和后续独立入口定义

### Agent 2：前台 UI 与视觉

负责：

- 首屏结构压缩
- 表单布局
- 视觉层级
- 结果区弱化

产出：

- 真正极简的 C 端首页

### Agent 3：生成链路与自动化逻辑

负责：

- 生成前自动选模板
- 可选主页链接自动参考
- 失败回退逻辑
- 自动卖点草稿

产出：

- 用户不用先操作模板，也能生成

### Agent 4：后台能力整理

负责：

- 把蒸馏、模板、筛片、高级控制重新归并到后台
- 让这些能力继续可用，但不打扰前台

产出：

- 官方高级后台区

### Agent 5：测试与验收

负责：

- 壳层测试
- 生成链路测试
- UI 走查
- Git 闭环

产出：

- 验证记录
- 提交说明

## 本轮明确不做

- 不接真实视频生成服务
- 不改付费、额度、账户权限逻辑
- 不改 TikTok / 抖音抓取算法本身
- 不拆出新后端

## 本轮验收标准

- 首页首屏只保留最少输入和主按钮。
- 普通用户不需要先理解模板和蒸馏，也能完成生成。
- 蒸馏、模板、筛片等能力仍保留，但明显退到后台位置。
- 生成前自动选择平台模板；如果填了主页链接，系统尽量自动参考该主页风格。
- 生成链路在主页参考失败时，不会卡死，而会自动退回默认模板。
- `node --check src/workspace.js` 通过。
- `node --test tests/extension-shell.test.mjs` 通过。
- `node --test tests/remake-core.test.mjs` 通过。

## 当前落地结果

- 首页已经从旧版分步骤工作台，改成 `单主任务大卡片`。
- `可选参考` 默认折叠，主页链接不再挡住主流程。
- `官方高级后台` 默认折叠，模板管理、主页蒸馏、样本筛片、参考视频和本地服务全部后置。
- 结果区已经强调 `最近生成` 和 `一键提交去跑视频`，不再让详细后台信息抢首屏。
- 平台语言继续自动判断：TikTok 默认英文，抖音默认中文。
- Chrome 当前已加载的扩展实例还是旧缓存页，所以这次验收以源码、测试和本地结构变更为准；用户重新加载扩展后即可看到新版前台。
