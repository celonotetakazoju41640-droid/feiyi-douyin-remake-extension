# 2026-07-17 生成页交互彻底重构

## 用户原始需求摘要
把 Chrome 扩展工作台里这条链路收完整：选择蒸馏模型 -> 一键传图 -> AI 自动识别产品卖点和场景 -> 自动生成提示词 -> 一键生成叙事图 -> 复制提示词和图片带走。前台尽量傻瓜式，复杂逻辑藏后台，细节要考虑到位。

## 这次我对需求的理解
这次不是继续局部补按钮或补提示，而是把“生成”页收成一个普通人第一次打开也能直接走完的主流程。重点不是新增业务能力，而是把已经有的识图、自动补草稿、故事版、一键带走这些能力，用更直白的页面结构重新串成一条主链路。

## 当前假设
- 真实目标仓库是 `/Users/da/Desktop/飞蚁抖音复刻-Chrome扩展`。
- 当前主要问题是前台交互结构混乱，不是后端主能力缺失。
- 用户要的是“直接能用的收口版”，不是再先出一版纯方案图。
- 当前工作区里 `src/workspace.css` 已有别的未提交改动，本次只增量兼容，不覆盖原有差异。

## 计划修改范围
- `src/workspace.html`
  - 重排生成页首屏，让主动作、系统自动动作、生成后去向更清楚。
- `src/workspace.js`
  - 统一生成页状态表达，补生成后结果承接、复制提示词、去历史记录快捷入口。
- `src/workspace.css`
  - 只追加与本轮生成页新结构直接相关的样式，不回滚现有未提交修改。
- `tests/extension-shell.test.mjs`
  - 补主链路交互结构和生成后承接的断言。
- `README.md`
  - 同步生成页的新使用口径。
- `docs/execution-plans/2026-07-17-生成页交互彻底重构.md`
  - 记录计划、验收和实际结果。

## 明确不修改的范围
- 不改商业规则、权限规则、资源消耗规则。
- 不接入新的真实付费服务或新的外部生成能力。
- 不重做“历史记录”和“蒸馏管理”的整体结构，只补与这条生成链路直接相关的承接口。
- 不处理当前仓库里与本次无关的未跟踪文件和其他人未提交改动。

## 执行步骤
1. 先补测试，锁定生成页应该出现的主链路结构、自动动作区、生成后承接区和快捷动作。
2. 重排 `workspace.html` 的生成页首屏，让输入区、系统自动动作区、生成后承接区分层清楚。
3. 更新 `workspace.js`，把识图中、草稿完成、可生成、已生成、可带走这些状态收成统一口径。
4. 只追加必要样式，让新结构在桌面和窄屏下都可读。
5. 同步 README 里的生成链路描述。
6. 跑测试、语法检查、diff 检查。
7. 提交 Git，并把实际结果补回本文件。

## 验收标准
- 新用户打开 `生成` 页，第一眼能看到：现在该做什么、系统会自动做什么、做完之后去哪里拿结果。
- 上传商品图后，页面会明确显示 AI 识别、自动补草稿和当前可否直接生成。
- 生成完成后，生成页本身就能给出：复制提示词、去历史记录看详情、继续一键带走结果的入口。
- `node --test tests/extension-shell.test.mjs` 通过。
- `node --check src/workspace.js`、`node --check src/remake-core.js` 通过。
- `git diff --check` 通过。

## 风险和需要注意的事
- 本次会涉及 `src/workspace.css`，但该文件已有未提交改动，所以只能追加兼容性样式，不能粗暴覆盖。
- 这次不触发真实外部视频生成，不会引入新的真实成本。

## 执行后的实际结果
- 已把生成页首屏改成“左边主动作 + 右边系统自动动作”的双区结构，用户第一眼就能分清自己要做什么、系统会自动做什么。
- 新增“系统会自动替你做这 4 件事”清单，明确显示识别商品图、拆卖点场景、补提示词草稿、承接故事版和带走结果这四段状态。
- 新增“这轮结果已经接住了”承接区，生成完成后可直接复制当前提示词草稿、继续一键带走全部结果，或跳去历史记录看完整结果。
- 一键主动作在 `autoDeliver` 场景下不再强行把用户踢到历史记录，而是优先留在生成页继续收口这条链路；普通“只生成项目”仍跳历史记录。
- README 和测试已同步更新。
