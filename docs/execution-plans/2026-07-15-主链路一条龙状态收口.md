# 2026-07-15 主链路一条龙状态收口

## 用户原始需求摘要

用户要继续把主链路收完整，不是再加新模块，而是把“选蒸馏模型 -> 传图 -> AI 自动识别商品卖点和场景 -> 生成提示词 -> 一键生成叙事图 -> 复制提示词和图片带走”这条线做得更傻瓜式、更清楚、更稳。

## AI 对需求的理解

当前能力已经基本接上，但普通用户仍然不一定一眼看懂：

- 现在走到哪一步
- 为什么按钮能点或不能点
- 故事版是不是已经开始跑、还要等多久
- 一键带走到底带走了哪些结果

这轮不扩功能，只收主链路状态表达和完成反馈，让前台更像“一路按下去就行”。

## 当前假设

- 故事版、本地导出、复制结果包的底层能力已可用，本轮不重写能力本身。
- 用户当前最在意的是状态清晰和页面可理解，不是再补新字段或新按钮。
- 现有 `src/workspace.css` 有无关脏改，本轮不能顺手处理样式文件。

## 计划修改范围

- `src/workspace.html`
- `src/workspace.js`
- `tests/extension-shell.test.mjs`
- `README.md`
- 本执行计划文档

## 明确不修改的范围

- 不改蒸馏管理页业务逻辑
- 不改 TikTok / 抖音提示词正文
- 不扩新功能
- 不碰 `scripts/runtime/`
- 不碰 `src/workspace-flow-test.html`
- 不处理无关的 `src/workspace.css` 既有脏改

## 执行步骤

1. 先补测试，锁定主流程状态条、故事版处理中提示、一键带走完成摘要、按钮与反馈一致性。
2. 修改 `src/workspace.html`，补最小可见的主流程状态容器。
3. 修改 `src/workspace.js`，统一生成态、故事版等待态、交付完成态和按钮口径。
4. 同步 `README.md`，写清当前一条龙流程对用户的实际反馈。
5. 跑检查命令，只提交本轮相关文件，完成 Git 上传闭环。

## 验收标准

- 生成页能看见极简主流程状态表达，用户知道当前卡在哪一步。
- 故事版处理中有稳定、可持续更新的进度反馈，不只靠一次性提示。
- 一键带走完成后，用户能看到复制 / 导出 / 故事版图片的完成摘要。
- 生成主按钮的禁用状态和页面反馈口径一致。
- `node --check src/workspace.js`
- `node --test tests/extension-shell.test.mjs`
- `git diff --check`
- `git diff --cached --check`

## 风险和需要注意的事项

- 不能为了补状态表达去改大块布局，否则容易误伤当前已可用页面。
- 一键带走会触发本地复制和导出，本轮只能把反馈说清楚，不能虚报“全部成功”。
- 如果故事版仍依赖本地服务返回，前台必须把“处理中 / 可继续等待 / 已完成 / 部分失败”区分清楚。
