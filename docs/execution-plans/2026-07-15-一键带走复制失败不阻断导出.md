# 2026-07-15 一键带走复制失败不阻断导出

## 用户原始需求摘要

用户要把“复制提示词图片一条龙”真正跑顺，不能因为其中一步小失败就整条链路断掉。

## AI 对需求的理解

代码审计发现当前 `runDeliveryShortcut()` 的顺序是：

1. 复制结果包
2. 导出文本和故事版图片

如果第 1 步因为剪贴板权限或系统限制失败，后面的导出根本不会执行。

这会让“一键带走结果”变成“先试复制，复制失败就整条停掉”，不符合一条龙交付预期。

## 当前假设

- 本轮只调整一键带走内部步骤的失败分流。
- 不触发真实外部服务。

## 计划修改范围

- `src/workspace.js`
- `tests/extension-shell.test.mjs`
- `README.md`
- 本执行计划文档

## 明确不修改的范围

- 不改故事版生成逻辑
- 不改真实生成链路
- 不碰 `src/workspace.css`
- 不碰 `scripts/runtime/`
- 不碰 `src/workspace-flow-test.html`

## 执行步骤

1. 补测试，锁定复制失败不能阻断导出。
2. 修改 `src/workspace.js`，把复制和导出拆成独立失败分支，并汇总反馈。
3. 同步 `README.md`。
4. 跑检查命令并完成 Git 闭环。

## 验收标准

- 复制结果包失败时，文本导出和故事版图片下载仍会继续尝试。
- 页面反馈能区分“复制失败但导出成功”和“导出失败”等情况。
- `node --check src/workspace.js`
- `node --test tests/extension-shell.test.mjs`
- `git diff --check`
- `git diff --cached --check`

## 风险和需要注意的事项

- 不能把“复制失败但导出成功”误报成“全部失败”。
- 不能把多个失败点混成一句无法判断的提示。
