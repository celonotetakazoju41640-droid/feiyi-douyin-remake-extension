# 2026-07-15 常驻状态刷新后不丢

## 用户原始需求摘要

用户要的是一条龙闭环，不只是当下能看懂，刷新后、切回项目后也要继续看得懂当前处在什么状态。

## AI 对需求的理解

代码审计发现当前 `当前故事版进度 / 当前带走结果` 这两类常驻状态只存在内存 `Map`：

- `projectWorkflowStatus`

页面一刷新，或者重新打开工作台后：

- 故事版进度摘要会丢
- 一键带走结果摘要会丢

这会让已经处理过一半的项目回到“像没做过一样”的感觉。

## 当前假设

- 本轮只补这两类状态的持久化和恢复。
- 不触发真实外部服务。

## 计划修改范围

- `src/workspace.js`
- `tests/extension-shell.test.mjs`
- `README.md`
- 本执行计划文档

## 明确不修改的范围

- 不改故事版生成逻辑
- 不改真实生成链路
- 不碰 `src/workspace.css`
- 不碰 `scripts/runtime/`
- 不碰 `src/workspace-flow-test.html`

## 执行步骤

1. 补测试，锁定 workflow 状态要写回项目并在加载时恢复。
2. 修改 `src/workspace.js`，让 `storyboardStatusSummary / deliveryStatusSummary` 跟项目一起保存和读取。
3. 同步 `README.md`。
4. 跑检查命令并完成 Git 闭环。

## 验收标准

- 刷新后，只要项目里已有故事版进度或带走结果摘要，前台仍能继续展示。
- 导入 JSON 或重新选中项目后，也能恢复这两类状态。
- `node --check src/workspace.js`
- `node --test tests/extension-shell.test.mjs`
- `git diff --check`
- `git diff --cached --check`

## 风险和需要注意的事项

- 不能把这类状态只放内存里，要真正写回项目数据。
- 不要污染核心导出结构，只加最小、可解释的状态字段。
